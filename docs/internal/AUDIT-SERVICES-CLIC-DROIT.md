# Audit: macOS Services (right-click) ‚Äì Status and fixes

**Date**: 2026-02-13  
**Context**: The user has Services entries (Ghosty ‚Äì Direct, etc.) in the right-click ‚Üí Services menu, but when launching a mode the system error appears: *"This service could not be launched because it is not configured correctly."*

---

## 1. Code and doc status

| Item | Status |
|------|--------|
| **Flag** | `ENABLE_RIGHT_CLICK_SERVICES = false` in `lib.rs` and `src/types/index.ts` |
| **Flag effect** | When `false`: the **deep link** `ghosty://` is not registered in the app (the `on_open_url` callback is not wired). So even if the workflow runs `open "ghosty://transform?mode=..."`, the app does not react. |
| **UI** | The Settings section "Install Services" / "Open Services folder" is hidden when the flag is `false`. |
| **Installed workflows** | Present in `~/Library/Services/` (Ghosty ‚Äì Direct.workflow, etc.), generated by `services_installer.rs`. |
| **Doc** | `docs/internal/SERVICES-CLIC-DROIT-MACOS.md` up to date (feature pending, flag at `false`). |

Conclusion: Workflows are created and visible in the menu, but two things can cause the error or incorrect behavior:
1. **Service configuration**: `Info.plist` used `NSRequiredContext` with `NSTextContent` = `"Text"`. Apple docs state that valid values for `NSTextContent` are: `URL`, `Date`, `Address`, `Email`, `FilePath` ‚Äî not `"Text"`. An invalid value can make the system reject the service ("not configured correctly").
2. **Deep link disabled**: Even if the service ran, `open ghosty://...` would not be handled by the app while `ENABLE_RIGHT_CLICK_SERVICES` is `false`.

---

## 2. Fixes applied

### 2.1 Info.plist (workflows)

- **Before**: `NSRequiredContext` with `NSTextContent` = `"Text"` (undocumented value for this field).
- **After**: `NSRequiredContext` left as empty dictionary `{}`. Apple docs: *"Always include this property, even when you do not require any filtering (in which case you specify an empty dictionary as the value)."* The received data type is still defined by `NSSendTypes` = `public.utf8-plain-text`, so the service still receives text.

### 2.2 Re-enabling the right-click flow

- **Rust**: `ENABLE_RIGHT_CLICK_SERVICES` set to `true` in `src-tauri/src/lib.rs`.
- **Front**: `ENABLE_RIGHT_CLICK_SERVICES` set to `true` in `src/types/index.ts`.
- Effect: registration of the handler for `ghosty://`, display of "Install Services" / "Open Services folder" settings, and handling of `run_transform_selection` when clicking a mode from Services.

### 2.3 Reinstall workflows after fixes

After updating the code, **reinstall** Services so workflows use the new `Info.plist`:

1. Launch Ghosty (release build from the .app).
2. Settings > System > **Install Services**.
3. System Settings > Keyboard Shortcuts > Services (Text): verify that **Ghosty ‚Äì ‚Ä¶** entries are checked.
4. Test: select text ‚Üí right-click ‚Üí Services ‚Üí Ghosty ‚Äì Direct (or another mode).

---

## 3. If the error persists

- **Console**: Open the **Console** app, filter by `WorkflowServiceRunner` or `Automator`, run the service again and check the error messages.
- **Manual workflow**: Create a Quick Action in Automator (receives Text, "Run Shell Script" with `open "ghosty://transform?mode=light"`), save it in `~/Library/Services/`, enable it in Shortcuts > Services. If that one works, the issue is with the format generated by `services_installer.rs`; if it also fails, the issue is elsewhere (ghosty:// scheme, app not launched from .app, etc.).
- **ghosty:// scheme**: The app must have been launched at least once from the **built .app** (not `npm run tauri dev`) for the scheme to be registered.
